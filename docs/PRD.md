# PRD و Tech Spec: ابزار هوشمند افزودن محصول به باسلام

تاریخ: ۱۴۰۵/۰۲/۱۰

## ۱. خلاصه محصول

این محصول یک ابزار SaaS چندغرفه‌دار برای غرفه‌داران باسلام است که افزودن چند محصول را در محیطی شبیه اکسل سریع‌تر و کم‌خطاتر می‌کند. کاربر چند عکس محصول را از گالری اضافه می‌کند؛ هر عکس به یک ردیف محصول تبدیل می‌شود؛ AI تصویر را تحلیل می‌کند، تصویر را به شکل محافظه‌کارانه بهبود می‌دهد، دسته‌بندی و ویژگی‌های لازم باسلام را پیشنهاد می‌کند، قیمت پیش‌فرض را از محصولات مشابه بازار باسلام استخراج می‌کند، و همه داده‌ها را در ستون‌های قابل مشاهده و قابل ویرایش نمایش می‌دهد.

محصول در نسخه اول هیچ انتشار خودکاری انجام نمی‌دهد. هر ردیف فقط بعد از بازبینی و اقدام صریح کاربر به باسلام ارسال می‌شود.

## ۲. مسئله و هدف

غرفه‌داران برای افزودن محصول باید عکس، عنوان، توضیحات، دسته، قیمت، موجودی، وزن، ارسال و ویژگی‌های دسته‌بندی را وارد کنند. این فرایند برای چند محصول تکراری، زمان‌بر و مستعد خطاست. هدف نسخه اول این است که کاربر با انتخاب چند عکس، پیش‌نویس چند محصول را در یک جدول قابل ویرایش دریافت کند و فقط موارد مشکوک یا ناقص را اصلاح کند.

اهداف اصلی:

- کاهش زمان ساخت چند محصول از عکس تا ارسال به باسلام.
- پر کردن خودکار دسته‌بندی و attributes بر اساس تصویر و داده‌های واقعی باسلام.
- پیشنهاد قیمت قابل ویرایش بر اساس محصولات مشابه در بازار باسلام.
- حفظ کنترل نهایی کاربر قبل از ارسال محصول.
- نمایش همه فیلدهای ارسالی به باسلام به شکل ستون‌های اکسل‌مانند.

غیرهدف‌های نسخه اول:

- انتشار یا ارسال محصول بدون تأیید کاربر.
- مدیریت سفارش، تخفیف، قفسه، کمپین یا موجودی پس از ساخت محصول.
- ویرایش محصولات موجود باسلام.
- قیمت‌گذاری قطعی یا تضمینی.
- تغییر ماهیت یا ظاهر واقعی محصول در تصویر بهبود‌یافته.

## ۳. کاربران و سناریوی اصلی

کاربر هدف، غرفه‌دار باسلامی است که چند محصول فیزیکی دارد و می‌خواهد آن‌ها را با کمترین ورود دستی وارد باسلام کند. سناریوی اصلی:

1. غرفه‌دار با OAuth باسلام وارد ابزار می‌شود.
2. چند عکس از گالری انتخاب می‌کند.
3. هر عکس به یک ردیف جدول تبدیل می‌شود.
4. سیستم برای هر ردیف تصویر را تحلیل و بهبود می‌دهد.
5. سیستم دسته‌بندی پیشنهادی و attributes مرتبط با دسته را از باسلام استخراج و پر می‌کند.
6. سیستم قیمت پیشنهادی را از محصولات مشابه بازار باسلام محاسبه می‌کند.
7. کاربر ستون‌ها را مرور و ویرایش می‌کند.
8. کاربر ردیف‌های آماده را انتخاب و ارسال می‌کند.
9. سیستم تصویر بهبود‌یافته را آپلود و محصول را در باسلام ایجاد می‌کند.
10. شناسه محصول ساخته‌شده و وضعیت نهایی در جدول نمایش داده می‌شود.

## ۴. تجربه کاربری

### ۴.۱ ورود و اتصال غرفه

- محصول باید OAuth باسلام را برای هر غرفه‌دار پشتیبانی کند.
- پس از ورود، سیستم باید `vendor_id` غرفه کاربر را از اطلاعات حساب استخراج کند.
- اگر کاربر غرفه فعال نداشته باشد، محصول نباید اجازه ایجاد محصول بدهد و باید پیام قابل فهم نشان دهد.
- توکن‌ها باید امن ذخیره شوند و refresh token برای تمدید نشست استفاده شود.

### ۴.۲ ورود عکس و ساخت ردیف

- کاربر بتواند چند عکس را هم‌زمان از گالری انتخاب کند.
- در MVP هر عکس یک محصول مستقل است.
- هر ردیف ابتدا با وضعیت `Image Added` ساخته شود.
- سیستم پردازش ردیف‌ها را مستقل انجام دهد تا خطای یک ردیف مانع پردازش بقیه نشود.
- کاربر بتواند ردیف را حذف، duplicate، و دوباره تحلیل کند.

### ۴.۳ جدول اکسل‌مانند

جدول باید:

- ویرایش inline سلول‌ها را پشتیبانی کند.
- ستون‌های ثابت و داینامیک داشته باشد.
- فیلدهای ناقص، کم‌اعتماد یا نامعتبر را مشخص کند.
- امکان مرتب‌سازی و فیلتر بر اساس وضعیت ردیف را داشته باشد.
- امکان انتخاب چند ردیف و ارسال batch را داشته باشد.
- خطاها را در سطح ردیف و سلول نمایش دهد.

### ۴.۴ کنترل کاربر قبل از ارسال

- دکمه ارسال فقط برای ردیف‌های `Ready` فعال باشد.
- اگر محصول attribute اجباری خالی، قیمت نهایی خالی، عکس آپلودنشده، دسته نامعتبر یا خطای validation داشته باشد، ارسال مسدود شود.
- سیستم باید قبل از ارسال، خلاصه تعداد ردیف‌های آماده، ناقص و دارای خطا را نشان دهد.

## ۵. ستون‌های جدول

### ۵.۱ ستون‌های ثابت

ستون‌های ثابت همه ردیف‌ها:

- عکس اصلی
- عکس بهبود‌یافته
- وضعیت ردیف
- دسته‌بندی
- اطمینان دسته‌بندی
- نام محصول
- توضیح کوتاه
- توضیح کامل
- کلمات کلیدی
- قیمت پیشنهادی
- قیمت نهایی
- منبع و اطمینان قیمت
- موجودی
- SKU
- وزن محصول
- وزن بسته‌بندی
- ابعاد بسته‌بندی
- زمان آماده‌سازی
- مقدار واحد
- نوع واحد
- روش ارسال
- عمده‌فروشی
- خطاها و هشدارها
- شناسه محصول ساخته‌شده

### ۵.۲ ستون‌های داینامیک attributes

بعد از تعیین `category_id` هر ردیف، سیستم باید endpoint زیر را فراخوانی کند:

```text
GET /v1/categories/{category_id}/attributes
```

خروجی attributes باید برای ساخت ستون‌های داینامیک همان ردیف استفاده شود. هر attribute حداقل شامل این اطلاعات است:

- `id`
- `title`
- `required`
- `type`
- `unit`
- `selected_values`

قواعد:

- عنوان ستون باید از `title` attribute ساخته شود.
- شناسه داخلی ستون باید `attribute:{id}` باشد.
- اگر `required=true` باشد، خالی بودن سلول باید مانع ارسال شود.
- اگر `selected_values` مقدار داشت، سلول باید به صورت انتخابی نمایش داده شود.
- اگر `unit` وجود داشت، واحد در header یا tooltip ستون نمایش داده شود.
- با تغییر دسته‌بندی، ستون‌های داینامیک ردیف باید بر اساس attributes دسته جدید refresh شوند.
- مقادیر attributes قبلی فقط زمانی نگه داشته شوند که attribute با همان `id` در دسته جدید هم وجود داشته باشد.

## ۶. AI Analysis

### ۶.۱ ورودی

ورودی تحلیل AI برای هر ردیف:

- تصویر اصلی محصول
- زبان پیش‌فرض فارسی
- لیست دسته‌بندی‌های باسلام از `GET /v1/categories`
- در صورت وجود، پیش‌فرض‌های غرفه برای موجودی، زمان آماده‌سازی، وزن بسته‌بندی، روش ارسال و عمده‌فروشی

### ۶.۲ خروجی ساختاریافته

AI باید خروجی ساختاریافته قابل parse تولید کند:

```json
{
  "title": "string",
  "brief": "string",
  "description": "string",
  "category": {
    "id": 0,
    "title": "string",
    "confidence": 0.0
  },
  "keywords": ["string"],
  "attributes": [
    {
      "attribute_id": 0,
      "title": "string",
      "value": "string",
      "confidence": 0.0
    }
  ],
  "estimated_weight": {
    "value": 0,
    "unit": "gram",
    "confidence": 0.0
  },
  "estimated_packaging_dimensions": {
    "length": 0,
    "width": 0,
    "height": 0,
    "confidence": 0.0
  },
  "warnings": ["string"]
}
```

قواعد:

- هر خروجی با confidence پایین باید در جدول علامت بخورد.
- AI نباید برای فیلدهای غیرقابل مشاهده، مقدار قطعی وانمود کند.
- وزن و ابعاد برآوردی فقط پیشنهاد هستند و در صورت confidence پایین باید `Needs Review` ایجاد کنند.
- قیمت توسط AI تصویر تعیین نمی‌شود و باید از الگوریتم قیمت بازار باسلام بیاید.

## ۷. AI Image Enhancement

بهبود تصویر باید محافظه‌کارانه باشد:

- اصلاح نور، contrast، sharpness و نویز مجاز است.
- crop و کادربندی محصول مجاز است.
- حذف یا ساده‌سازی پس‌زمینه مجاز است، به شرطی که محصول تغییر نکند.
- افزودن اجزای جدید، تغییر رنگ محصول، تغییر جنس، تغییر اندازه ظاهری فریبنده یا تولید نسخه غیرواقعی محصول ممنوع است.
- تصویر اصلی همیشه باید کنار تصویر بهبود‌یافته قابل مشاهده باشد.
- کاربر باید بتواند برای هر ردیف تصویر اصلی یا تصویر بهبود‌یافته را برای ارسال انتخاب کند.

## ۸. دسته‌بندی و attributes باسلام

### ۸.۱ دریافت دسته‌ها

سیستم باید در شروع نشست یا طبق cache policy، درخت دسته‌بندی را از باسلام بگیرد:

```text
GET /v1/categories
```

خروجی شامل `id`, `title`, `children` و در برخی دسته‌ها `unit_type` است.

### ۸.۲ انتخاب خودکار دسته

AI باید فقط از بین دسته‌های معتبر باسلام دسته انتخاب کند. اگر confidence دسته کمتر از آستانه باشد، ردیف باید `Needs Review` شود.

آستانه پیش‌فرض:

- `category_confidence >= 0.75`: قابل قبول
- `0.50 <= category_confidence < 0.75`: نیازمند بازبینی
- `< 0.50`: دسته انتخاب نشده تلقی شود

### ۸.۳ تغییر دستی دسته

کاربر باید بتواند دسته را دستی جستجو و انتخاب کند. بعد از انتخاب دستی:

- attributes دسته جدید واکشی شود.
- ستون‌های attribute ردیف refresh شود.
- فیلدهای مربوط به unit type دسته در صورت وجود پیشنهاد شود.
- وضعیت validation دوباره محاسبه شود.

## ۹. قیمت پیشنهادی از بازار باسلام

قیمت پیش‌فرض از محصولات خود غرفه استخراج نمی‌شود. منبع قیمت فقط بازار باسلام است.

### ۹.۱ API جستجو

سیستم باید از جستجوی محصولات باسلام استفاده کند:

```text
POST /v1/products/search
```

ورودی جستجو:

- `q`: عنوان پیشنهادی AI و کلمات کلیدی اصلی
- `rows`: حداقل ۱۲ و ترجیحا ۲۴ نتیجه
- `filters`: در صورت امکان شامل slug یا فیلتر دسته‌بندی مرتبط

### ۹.۲ فیلتر نتایج مشابه

بعد از دریافت نتایج:

- محصولات ناموجود یا غیرقابل فروش حذف شوند.
- محصولات با `new_categoryId` یا `categoryId` نامرتبط حذف شوند.
- محصولاتی که قیمت معتبر ندارند حذف شوند.
- اگر وزن یا `mainAttribute` قابل مقایسه بود، محصولاتی با واحد خیلی متفاوت جداگانه علامت بخورند.
- outlierها حذف شوند. روش پیش‌فرض: حذف قیمت‌های خارج از بازه `Q1 - 1.5*IQR` تا `Q3 + 1.5*IQR`.

### ۹.۳ محاسبه قیمت

- قیمت پیشنهادی برابر median مقدار `primaryPrice` نتایج فیلترشده باشد.
- اگر `primaryPrice` نبود، از `price` استفاده شود.
- خروجی قیمت شامل این فیلدها باشد:
  - `suggested_price`
  - `sample_count`
  - `min_price`
  - `max_price`
  - `median_price`
  - `confidence`
  - `source = "basalam_market_search"`

### ۹.۴ اطمینان قیمت

قواعد پیش‌فرض confidence:

- `sample_count >= ۸` و دسته‌ها همسان: confidence بالا
- `۴ <= sample_count < ۸`: confidence متوسط
- `sample_count < ۴`: confidence پایین و ردیف `Needs Review`
- اگر نتایج هم‌دسته کافی نبود، قیمت پیشنهادی خالی بماند و هشدار نمایش داده شود.

قیمت پیشنهادی همیشه قابل ویرایش است و مقدار ارسال‌شده به باسلام فقط `قیمت نهایی` است.

## ۱۰. پیش‌فرض‌های غرفه

برای فیلدهایی که از عکس قابل تشخیص قابل اتکا نیستند، سیستم باید از پیش‌فرض‌های قابل تنظیم غرفه استفاده کند:

- موجودی پیش‌فرض
- زمان آماده‌سازی
- وزن بسته‌بندی
- ابعاد بسته‌بندی
- روش‌های ارسال
- عمده‌فروشی
- سیاست انتخاب تصویر اصلی یا بهبود‌یافته

اگر پیش‌فرض لازم موجود نباشد، ردیف باید `Needs Review` بماند.

## ۱۱. ادغام با API باسلام

### ۱۱.۱ احراز هویت

محصول باید OAuth Authorization Code Flow را برای SaaS چندغرفه‌دار پیاده‌سازی کند. scopeهای لازم حداقل باید دسترسی خواندن اطلاعات کاربر/غرفه، خواندن دسته‌ها، آپلود فایل و نوشتن محصول را پوشش دهند. توکن‌ها باید encrypted at rest ذخیره شوند.

### ۱۱.۲ آپلود تصویر

برای هر ردیف آماده، قبل از ساخت محصول تصویر انتخاب‌شده باید آپلود شود:

```text
POST /v1/files
Content-Type: multipart/form-data
file_type=product.photo
```

شناسه فایل پاسخ در فیلدهای `photo` و `photos` محصول استفاده شود. شناسه `photo` باید اولین آیتم `photos` هم باشد.

### ۱۱.۳ ساخت محصول

بعد از آپلود تصویر، محصول با endpoint زیر ساخته شود:

```text
POST /v1/vendors/{vendor_id}/products
```

mapping فیلدها:

- `name`: نام محصول
- `photo`: شناسه عکس اصلی
- `photos`: لیست شناسه عکس‌ها
- `brief`: توضیح کوتاه
- `description`: توضیح کامل
- `category_id`: دسته‌بندی انتخاب‌شده
- `preparation_days`: زمان آماده‌سازی
- `keywords`: کلمات کلیدی
- `weight`: وزن محصول
- `package_weight`: وزن بسته‌بندی
- `primary_price`: قیمت نهایی
- `stock`: موجودی
- `shipping_method_ids`: روش‌های ارسال
- `attributes` یا `product_attribute`: ویژگی‌های دسته، مطابق قرارداد فعال API/SDK باسلام
- `unit_quantity`: مقدار واحد
- `unit_type`: نوع واحد
- `sku`: SKU
- `packaging_dimensions`: ابعاد بسته‌بندی
- `is_wholesale`: عمده‌فروشی

مقدار `status` نباید از نمونه‌های مستندات hardcode شود. تیم توسعه باید enum معتبر وضعیت محصول را از SDK/تنظیمات باسلام استخراج و در config نگه دارد. در MVP ارسال فقط بعد از تأیید کاربر انجام می‌شود.

## ۱۲. وضعیت‌های ردیف

وضعیت‌های مجاز:

- `Image Added`: عکس اضافه شده ولی تحلیل شروع نشده است.
- `Analyzing`: تحلیل AI، بهبود تصویر، دسته‌بندی یا قیمت‌گذاری در حال انجام است.
- `Needs Review`: ردیف داده ناقص، کم‌اعتماد یا نیازمند ویرایش دارد.
- `Ready`: ردیف همه validationهای لازم برای ارسال را پاس کرده است.
- `Submitting`: آپلود تصویر یا ساخت محصول در حال انجام است.
- `Submitted`: محصول در باسلام ساخته شده و شناسه محصول ثبت شده است.
- `Failed`: ارسال یا پردازش شکست خورده و خطا قابل مشاهده است.

هر تغییر سلول باید validation ردیف را دوباره اجرا کند و در صورت رفع همه خطاها وضعیت را به `Ready` برگرداند.

## ۱۳. Validation و Edge Cases

مواردی که باید پوشش داده شود:

- دسته با confidence پایین: ردیف `Needs Review` و سلول دسته highlighted شود.
- attribute اجباری خالی: ارسال مسدود شود.
- قیمت مشابه ناکافی: قیمت پیشنهادی خالی یا کم‌اعتماد و ردیف `Needs Review` شود.
- خطای آپلود تصویر: ردیف `Failed` شود و retry فقط برای همان ردیف ممکن باشد.
- خطای ساخت محصول ۴۲۲: خطاهای field-level در سلول‌های متناظر نمایش داده شود.
- خطای ۴۰۱: refresh token امتحان شود؛ اگر شکست خورد، کاربر دوباره وارد شود.
- timeout یا rate limit: retry با backoff و نمایش وضعیت ردیف.
- تغییر دسته بعد از پر شدن attributes: ستون‌های قبلی نامعتبر پاک یا غیرفعال شوند.
- محصول بدون قیمت نهایی، عکس، دسته یا موجودی قابل ارسال نباشد.
- اگر کاربر تصویر بهبود‌یافته را رد کرد، تصویر اصلی برای آپلود استفاده شود.

## ۱۴. داده‌ها و مدل مفهومی

موجودیت‌های اصلی:

- `User`: کاربر SaaS و اطلاعات اتصال OAuth.
- `VendorConnection`: اتصال کاربر به غرفه باسلام شامل `vendor_id` و tokenهای رمزنگاری‌شده.
- `ImportSession`: یک نشست ورود چند محصول.
- `ProductRow`: یک ردیف محصول شامل عکس، فیلدهای ثابت، attributes، وضعیت و خطاها.
- `AIAnalysisResult`: خروجی ساختاریافته AI و confidenceها.
- `MarketPriceSuggestion`: خروجی قیمت‌گذاری از جستجوی بازار باسلام.
- `BasalamSubmission`: نتیجه آپلود عکس و ساخت محصول در باسلام.

برای هر ردیف باید audit trail حداقلی نگه‌داری شود: زمان ساخت، زمان آخرین تحلیل، کاربر ویرایشگر، آخرین خطای API و شناسه محصول ساخته‌شده.

## ۱۵. امنیت و حریم خصوصی

- توکن‌های OAuth باید encrypted at rest باشند.
- تصاویر کاربران باید فقط برای تحلیل، بهبود و ارسال محصول استفاده شوند.
- دسترسی به ImportSession باید محدود به مالک همان اتصال غرفه باشد.
- لاگ‌ها نباید access token، refresh token یا فایل خام حساس را ذخیره کنند.
- برای فراخوانی AI باید retention و privacy policy سرویس AI بررسی و در محصول شفاف شود.

## ۱۶. معیارهای موفقیت

معیارهای MVP:

- حداقل ۷۰٪ ردیف‌ها بعد از تحلیل اولیه به `Ready` یا `Needs Review` با کمتر از ۳ فیلد ناقص برسند.
- زمان متوسط ساخت پیش‌نویس برای ۱۰ عکس کمتر از ۲ دقیقه باشد.
- حداقل ۸۰٪ ردیف‌های `Ready` بدون خطای ۴۲۲ در باسلام ساخته شوند.
- کاربر بتواند برای هر ردیف علت `Needs Review` را بدون ابهام ببیند.
- قیمت پیشنهادی حداقل با sample count و بازه قیمت قابل توضیح باشد.

## ۱۷. تست و پذیرش

سناریوهای تست:

- انتخاب چند عکس باید چند ردیف مستقل بسازد و هر ردیف ستون‌های ثابت داشته باشد.
- بعد از تعیین دسته، attributes همان دسته باید به ستون‌های داینامیک تبدیل شوند.
- تغییر دستی دسته باید attributes را refresh و validation را دوباره اجرا کند.
- قیمت پیشنهادی باید فقط از جستجوی بازار باسلام محاسبه شود و `sample_count`, `min_price`, `max_price`, `confidence` را نشان دهد.
- محصول بدون attribute اجباری، قیمت نهایی یا عکس قابل ارسال نباشد.
- آپلود تصویر با `file_type=product.photo` و استفاده از شناسه فایل در `photo` و `photos` تست شود.
- ساخت محصول با mock API و یک تست integration روی sandbox/محیط مجاز باسلام تست شود.
- خطاهای ۴۰۱، ۴۲۲، timeout و rate limit باید در سطح ردیف نمایش داده و retryپذیر باشند.
- اگر کاربر تصویر بهبود‌یافته را نپذیرد، تصویر اصلی باید آپلود شود.

## ۱۸. ریسک‌ها و تصمیم‌های قطعی

ریسک‌ها:

- تشخیص اشتباه دسته یا attribute می‌تواند باعث رد محصول در باسلام شود.
- قیمت بازار برای محصولات خاص یا دست‌ساز ممکن است نمونه کافی نداشته باشد.
- بهبود تصویر اگر کنترل نشود می‌تواند محصول را غیرواقعی نشان دهد.
- تفاوت نسخه‌های API/SDK باسلام می‌تواند mapping مربوط به `attributes` و `product_attribute` را تحت تأثیر قرار دهد.

تصمیم‌های قطعی نسخه اول:

- محصول SaaS چندغرفه‌دار با OAuth است.
- ورود محصول در MVP چند محصول جدولی است و هر عکس یک ردیف محصول می‌سازد.
- بهبود تصویر محافظه‌کارانه است.
- ارسال محصول فقط با تأیید کاربر انجام می‌شود.
- قیمت پیش‌فرض فقط از بازار باسلام استخراج می‌شود.
- همه attributes لازم باسلام از `GET /v1/categories/{category_id}/attributes` ساخته می‌شوند.
